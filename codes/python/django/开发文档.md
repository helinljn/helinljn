# 游戏GM命令后台管理系统 - 开发文档

## 一、项目概述

基于现有 Django 5.2.12 + Python 3.11 + SQLite 环境，开发一个游戏 GM 命令后台管理系统。系统实现基于用户级别的命令权限控制，每个用户的 GM 命令权限单独分配；角色分组仅用于用户分类标记，不承载权限。超级管理员拥有最高权限，可管理用户及权限分配。所有 GM 指令发送请求时，以 HTTP POST 形式转发至指定的 API 地址进行后续处理。

### 技术栈

| 项目 | 技术 |
|------|------|
| 后端框架 | Django 5.2.12 |
| 数据库 | SQLite（开发环境） |
| 前端样式 | Tabler（CDN，基于 Bootstrap 5） |
| 图标库 | Tabler Icons（CDN） |
| Python | 3.11 |
| HTTP 请求 | requests 库 |

### 4.2 权限控制流程

```
用户请求 → 是否登录？
              ├── 否 → 跳转登录页
              └── 是 → 是否超级管理员？
                          │  (Django is_superuser 或 GM super_admin 角色)
                          ├── 是 → 全部权限 + 管理功能
                          └── 否 → 用户是否有该命令的直接权限？
                                      │  (通过 UserCommandPermission 查询)
                                      ├── 是 → 允许执行命令
                                      └── 否 → 返回403禁止
```

### 4.3 超级管理员绑定机制

Django 内置超级管理员（`is_superuser=True`）与 GM 系统超级管理员（`super_admin` 角色）唯一绑定：

```
Django createsuperuser
       │
       ▼ post_save 信号触发
┌──────────────────────────────────┐
│ 1. 确保 super_admin 角色存在      │
│ 2. 创建/更新 UserProfile 绑定角色 │
│ 3. 自动授予所有活跃命令权限        │
└──────────────────────────────────┘

migrate 执行完成
       │
       ▼ post_migrate 信号触发
┌──────────────────────────────────┐
│ 为所有已有的 Django 超级管理员     │
│ 补充绑定 GM super_admin 角色      │
└──────────────────────────────────┘
```

## 五、IDIP API 转发协议

### 5.1 请求格式

```
POST {IDIP_API_URL}
Content-Type: application/x-www-form-urlencoded

id=10226000&GSA=&content=%7B%22head%22%3A...%7D
```

### 5.2 参数说明

| 参数 | 说明 | 示例 |
|------|------|------|
| `id` | 命令编号 | `10226000` |
| `GSA` | 固定为空 | `` |
| `content` | JSON 字符串经 URL 编码后的值 | `%7B%22head%22...%7D` |

### 5.3 content 字段 JSON 结构

```json
{
  "head": {
    "Uin": 0,
    "AreaId": 10,
    "Partition": 1201,
    "PlatId": 1,
    "Requestid": 4097,
    "Request": "QueryUsrInfoReq",
    "Responseid": 4098,
    "Response": "QueryUsrInfoRsp"
  },
  "body": {
    "OpenId": "xxx",
    "RoleId": "123",
    "RoleName": ""
  }
}
```

### 5.4 请求构造逻辑（伪代码）

```python
import json
import urllib.parse
import requests

def send_idip_command(command, params):
    content_json = {
        "head": {
            "Uin": 0,
            "AreaId": params.get("AreaId", 10),
            "Partition": params["Partition"],
            "PlatId": params.get("PlatId", 1),
            "Requestid": command.request_id,
            "Request": command.request_name,
            "Responseid": command.response_id,
            "Response": command.response_name,
        },
        "body": {k: v for k, v in params.items()
                 if k not in ("AreaId", "Partition", "PlatId")}
    }

    content_encoded = urllib.parse.quote(json.dumps(content_json, ensure_ascii=False))
    form_data = {
        "id": command.command_id,
        "GSA": "",
        "content": content_encoded,
    }

    # 构造完整请求JSON，用于存储到数据库和日志记录
    full_request = {
        "url": settings.IDIP_API_URL,
        "method": "POST",
        "form_data": form_data,
        "content_json": content_json,
    }
    request_content_str = json.dumps(full_request, ensure_ascii=False)

    # 记录请求发送日志
    logger.info('IDIP请求发送: command=%s, content=%s', command.command_id, ...)

    try:
        response = requests.post(
            settings.IDIP_API_URL,
            data=form_data,  # form-urlencoded
            timeout=settings.IDIP_TIMEOUT,
        )
        response.raise_for_status()
        result = response.json()
        logger.info('IDIP请求成功: command=%s, response=%s', command.command_id, ...)
        return result, None, request_content_str
    except requests.Timeout:
        logger.error('IDIP API请求超时: command=%s', command.command_id)
        return None, '请求超时', request_content_str
    except requests.ConnectionError:
        logger.error('IDIP API连接失败: url=%s', settings.IDIP_API_URL)
        return None, '无法连接', request_content_str
    except Exception as e:
        logger.error('IDIP API请求异常: %s', e)
        return None, f'请求失败: {e}', request_content_str
```

> 返回值为三元组 `(response_data, error, request_content_str)`，无论成功或失败，`request_content_str` 始终包含完整请求JSON，用于写入 `CommandLog.request_content` 字段。

| 字段 | 类型 | 说明 |

| 字段 | 类型 | 说明 |

## 七、命令解析引擎设计

### 7.1 解析逻辑

解析 `idip_commands.json` 文件，提取每个命令的元数据和参数定义，存入 `GMCommand` 模型：

1. 读取 JSON 文件，遍历所有命令定义
2. 提取命令 ID、名称、tab 分组、请求/响应名称和 ID
3. 提取 `request.params` 列表作为 `request_params`（JSONField 存储）
4. 提取 `response.params` 列表作为 `response_params`（JSONField 存储）
5. 通过管理命令 `sync_commands` 执行同步（支持增量更新）

### 7.2 同步逻辑（`sync_commands_to_db()`）

将 JSON 中的命令定义同步到数据库，并**自动授权新增命令给超级管理员用户**：

| 操作 | 说明 |
|------|------|
| 新增 | JSON 中有、数据库无的命令 → 创建记录 + 授予超级管理员用户权限 |
| 更新 | JSON 和数据库都有的命令 → 更新字段内容 |
| 停用 | 数据库有、JSON 中无的命令 → 标记 `is_active=False` |

返回 `(created_count, updated_count, deactivated_count)`。

### 7.3 前端动态表单渲染规则

根据 `request_params` 中的参数类型定义，前端动态生成表单控件：

| 参数 type | 表单控件 | 说明 |
|-----------|---------|------|
| `string` | 文本输入框 `<input type="text">` | 普通文本 |
| `uint32` / `uint64` / `int32` / `uint8` | 数字输入框 `<input type="number">` | 整数类型 |
| `time` | 日期时间选择器 `<input type="datetime-local">` | 时间类型 |
| `isnull: "true"` | 非必填标记 | 字段可留空，标签显示"可选" |
| `multitarget: true` | 多目标输入 textarea | 支持批量（逗号/换行分隔） |
| `size` | maxlength 限制 | 输入长度限制 |

图标体系：使用 Tabler Icons（`ti ti-xxx`），不再使用 Bootstrap Icons。

## 九、视图函数设计

### 9.1 认证相关

- `login_view(request)` — 处理用户登录，验证用户名密码，设置 session。已认证用户访问时显示"已登录"提示页（含"进入系统"和"切换账号"按钮），避免静默跳转导致的安全疑虑
- `logout_view(request)` — 清除 session，重定向到登录页。已认证时记录登出日志，未认证时（如 session 已被其他标签页销毁）跳过日志记录避免报错

### 9.2 仪表盘

- `dashboard(request)` — 展示命令分类（按 tab 分组）、快捷入口、最近操作日志。超级管理员额外显示"命令配置管理"卡片（上传JSON文件+手动同步）

### 9.3 命令管理

- `command_list(request)` — 根据用户权限过滤显示可用命令列表
- `command_execute(request, cmd_id)` — GET 显示动态表单；POST 构造请求体并转发到 IDIP API，记录日志，展示返回结果。**并发保护**：若命令在用户填写表单期间被管理员停用/删除，POST 请求返回 `410 Gone` + `{"command_deactivated": True}`，前端显示提示后自动跳转命令列表；GET 请求则通过 `messages` 提示并重定向

### 9.4 用户管理（super_admin）

- `user_list(request)` — 用户列表，支持搜索，显示权限数量
- `user_create(request)` — 创建用户表单
- `user_edit(request, user_id)` — 编辑用户信息、角色分组；**禁止用户禁用自己的账号**（前端禁用复选框 + 后端双重校验）
- `user_delete(request, user_id)` — 删除用户
- `user_permissions(request, user_id)` — 为用户单独分配命令权限（复选框勾选，超级管理员无需分配自动拥有全部权限）

### 9.5 角色分组管理（super_admin）

- `role_list(request)` — 角色分组列表
- `role_create(request)` — 创建角色分组
- `role_edit(request, role_id)` — 编辑角色分组信息
- `role_delete(request, role_id)` — 删除角色分组

### 9.6 日志

- `command_log_list(request)` — 分页展示命令执行日志，支持按用户/命令/状态筛选。请求参数列过长时缩略显示，点击可弹出详情弹窗查看完整请求JSON、表单参数和响应数据
- `login_log_list(request)` — 分页展示登录日志（仅超管），支持按用户名/操作类型筛选

### 9.7 API

## 十、权限装饰器设计

### 10.1 `is_super_admin(user)` — 超级管理员判断

判断用户是否为超级管理员，**优先检查 Django 内置 `is_superuser`**，同时检查 GM 系统角色。两者任一为 `True` 即视为超级管理员，确保唯一绑定：

```python
def is_super_admin(user):
    if not user.is_authenticated:
        return False
    if user.is_superuser:          # Django 超管 → 直接通过
        return True
    try:
        profile = user.userprofile
        return profile.role is not None and profile.role.is_super_admin
    except UserProfile.DoesNotExist:
        return False
```

### 10.2 `login_required` 装饰器

检查用户是否已登录，未登录重定向到登录页。可使用 Django 内置的 `login_required`。

### 10.3 `super_admin_required` 装饰器

检查当前用户是否为超级管理员角色（含 Django `is_superuser`），不是则重定向到仪表盘。

### 10.4 `command_permission_required` 装饰器

检查当前用户是否拥有指定命令的执行权限，超级管理员（Django `is_superuser` 或 GM 角色）自动通过。普通用户通过 `UserCommandPermission` 直接查询用户级权限。

### 10.5 `get_user_permissions(user)` — 获取用户权限

返回用户可执行的命令 ID 列表。超级管理员返回所有活跃命令；普通用户从 `UserCommandPermission` 读取直接分配的权限。

### 16.1 模块说明

`audit_log.py` 提供统一的操作审计日志接口 `log_operation()`，将所有用户操作写入本地日志文件，与数据库中的 `CommandLog`（仅记录命令执行）互补。

### 16.2 `log_operation(category, action, user, detail, ip_address)`

| 参数 | 类型 | 说明 |
|------|------|------|
| `category` | str | 操作分类：`auth` / `user` / `role` / `permission` / `command` |
| `action` | str | 操作动作：`login` / `logout` / `login_failed` / `create` / `update` / `delete` / `assign` / `execute` / `sync` |
| `user` | User | 操作用户对象，未登录时为 `None` |
| `detail` | dict/str | 操作详情，自动序列化为 JSON |

### 16.3 日志格式

```

[2026-04-10 20:31:02] [user] create | user=admin | ip=192.168.1.100 | {"target_user":"test","is_active":true,"role":"operator"}
```

### 16.4 审计日志覆盖的操作

| 分类 | 动作 | 触发点 | 数据库记录 |
|------|------|--------|-----------|
| auth | login | 用户登录成功 | LoginLog + 审计文件 |
| auth | login_failed | 用户登录失败（含原因：账号禁用/密码错误） | LoginLog + 审计文件 |
| auth | logout | 用户登出 | LoginLog + 审计文件 |
| command | execute | 执行 GM 命令 | CommandLog + 审计文件 |
| command | sync | 手动同步命令定义 | 审计文件 |
| command | upload_and_sync | 上传JSON文件并同步命令定义 | 审计文件 |
| user | create | 创建用户 | 审计文件 |
| user | update | 编辑用户信息/角色 | 审计文件 |

| role | create | 创建角色 | 审计文件 |

| role | delete | 删除角色 | 审计文件 |
| permission | assign | 分配用户命令权限（含新增/移除的命令ID） | 审计文件 |

### 16.5 日志文件配置

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 日志文件路径 | `logs/audit.log` | 审计日志 |
| 应用日志路径 | `logs/gmtool.log` | 应用运行日志 |
| 轮转策略 | 每天午夜轮转 | `SafeTimedRotatingFileHandler(when='midnight')` |
| 审计日志保留 | 90 天 | `backupCount=90` |
| 应用日志保留 | 30 天 | `backupCount=30` |

### 16.6 SafeTimedRotatingFileHandler 设计

自定义 `TimedRotatingFileHandler`，解决 Windows 平台多线程/多进程下日志轮转的文件锁定问题。

**问题背景：** Windows 上文件被占用时 `os.rename()` 会抛出 `PermissionError`，而标准库 `doRollover()` 在关闭文件流后调用 `rotate()`，若重命名失败则文件流不会被重新打开，导致后续日志写入全部丢失。

**解决方案：** 完全重写 `doRollover()`，核心保证：

- `rotate()` 失败时仅跳过文件重命名，不阻断后续流程
- 始终重新打开 `self.stream`，确保日志写入不中断
- 始终更新 `rolloverAt`，避免每次 `emit` 重复触发轮转
- 清理旧备份时也容错处理

Linux 不存在此问题（POSIX 允许 rename 已打开的文件），上述容错逻辑不会影响正常行为。

## 十九、备注

1. **IDIP API 地址**：在 `settings.py` 中配置为 `IDIP_API_URL`，部署时根据实际环境修改。
2. **默认角色**：系统初始化仅创建超级管理员角色，其他角色分组由超级管理员在界面中创建。角色分组仅用于用户分类标记，不承载权限。
3. **权限分配方式**：GM 命令权限按用户单独分配（通过 `UserCommandPermission` 模型），在用户管理界面为每个用户设置可执行的命令权限。超级管理员自动拥有全部权限，无需手动分配。
4. **命令同步**：当 `idip_commands.json` 更新时，可通过以下方式同步：
   - 管理命令：`python manage.py sync_commands`
   - API 接口：`/gmtool/api/commands/sync/`（手动同步服务器上现有文件）
   - 上传接口：`/gmtool/api/commands/upload/`（上传JSON文件并自动同步）
   - **自动同步**：文件监控中间件检测到 JSON 文件变更后自动同步，新增命令自动授权给超级管理员用户
5. **超级管理员绑定**：Django `is_superuser=True` 的用户自动与 GM `super_admin` 角色唯一绑定，通过信号处理器实现：
   - 创建/保存超管用户时自动创建 UserProfile + 绑定角色 + 授予所有用户级权限
   - 数据库迁移后自动补绑已有的 Django 超管用户
6. **日志体系**：系统包含两层日志：
   - **数据库日志**：
     - `CommandLog`：记录命令执行详情（含完整请求JSON `request_content` 字段），所有登录用户可在操作日志页面查看，请求参数过长时缩略展示，点击请求参数文本或查看按钮弹出详情弹窗，展示完整请求JSON、表单参数和响应数据
     - `LoginLog`：记录登录/登出/登录失败，仅超管可在登录日志页面查看
   - **本地文件日志**（`logs/audit.log`）：记录所有用户操作审计日志（含登录/登出/用户CRUD/角色CRUD/权限变更/命令同步），每天午夜轮转，保留 90 天
   - **本地应用日志**（`logs/gmtool.log`）：记录应用运行日志（含IDIP请求发送/成功/失败的详细日志），每天午夜轮转，保留 30 天
7. **安全性**：生产环境建议关闭 Django DEBUG 模式，配置 ALLOWED_HOSTS，启用 HTTPS。
8. **缓存**：开发环境使用 `LocMemCache`，生产环境建议替换为 Redis 或 Memcached。
9. **文件监控节流**：中间件每 30 秒最多检查一次文件变更，避免频繁磁盘 I/O。
10. **自我保护机制**：用户编辑自己时，`is_active` 复选框自动禁用（`UserEditForm` 通过 `is_self` 参数控制），后端同步校验，防止误操作锁死自己的账号。删除用户时同样不允许删除自己。
11. **自定义错误页面**：系统提供美观的错误页面（403/404/500），避免暴露 Django 调试信息。`DEBUG=True` 时通过 URL 兜底模式生效，`DEBUG=False` 时通过 Django handler 机制生效。同时配置了 `CSRF_FAILURE_VIEW` 自定义处理：登录页 CSRF 失败时统一重定向回登录页并提示"页面已过期"，不调用 `auth_logout`（避免销毁共享 session），也不因已认证状态而放行（避免绕过密码验证）。已认证用户访问登录页时显示"已登录"提示，提供"进入系统"和"切换账号"两个选项，而非静默跳转。
